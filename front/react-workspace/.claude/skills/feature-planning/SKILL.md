# Feature Planning Workflow

아이디어에서 개발 가능한 이슈 단위로 변환하는 기획 파이프라인.
각 GATE에서 사용자의 명시적 승인을 받아야 다음 단계로 진행한다. 승인 없이 자동으로 앞서 나가지 않는다.

---

## 브랜치 기반 경로 자동 추론 (핵심 변경)

### 입력 형식

```
/feature-planner                          ← 현재 브랜치에서 자동 추론
/feature-planner {기능 설명}              ← 현재 브랜치 + 설명 추가
/feature-planner {feature-path}           ← 경로 직접 지정 (기존 방식)
/feature-planner {feature-path} {설명}   ← 경로 + 설명 (기존 방식)
```

### 브랜치 → feature-path 자동 파싱

`/feature-planner`만 입력하거나 첫 토큰이 `{설명}`처럼 보일 때 (슬래시 없고 한글/자연어인 경우),
아래 순서로 `feature-path`를 결정한다.

**Step 1: 현재 브랜치 확인**
```bash
git branch --show-current
```

**Step 2: 보호 브랜치 감지 → 브랜치 생성 프로세스**

브랜치명이 `main`, `master`, `develop`, `dev` 중 하나이거나,
`feature/` prefix가 없는 경우 → **기획 전에 브랜치를 먼저 만든다.**

```
⚠️  현재 브랜치: main (보호 브랜치)
    기획 문서와 이후 코드 작업은 feature 브랜치에서 진행해야 합니다.

    기능명을 알려주시면 브랜치명을 제안해드릴게요.
    예) "공지 목록"       → feature/notice/list
        "태그 관리"       → feature/tag
        "공지 검색 조건"  → feature/notice/list-search
```

사용자가 기능명을 입력하면:

1. **브랜치명 제안**
   ```
   제안 브랜치명: feature/{feature-path}
   예) feature/notice/list

   ✅ 이 이름으로 생성할까요?
      변경하려면 원하는 이름을 입력해주세요.
   ```

2. **사용자 확인 후 브랜치 생성 명령어 제시**
   > Claude는 git 명령을 직접 실행하지 않는다. 사용자가 직접 실행하도록 명령어를 제시한다.
   ```bash
   git checkout -b feature/notice/list
   # 또는 이미 원격에 브랜치가 있다면:
   git checkout -t origin/feature/notice/list
   ```
   ```
   위 명령어를 실행하신 후 "완료"라고 말씀해주세요.
   브랜치 전환이 확인되면 기획을 시작합니다.
   ```

3. **사용자가 "완료" 응답 시**
   - `feature-path`를 브랜치명에서 추론해 컨텍스트에 저장
   - 경로 확인 메시지 출력 후 spec.md 처리로 진행

**Step 3: 브랜치 → feature-path 변환 규칙**

> **브랜치 컨벤션**
> - `/` : 계층 구분자 (도메인/기능/하위기능)
> - `-` : 같은 계층 내 단어 연결
>
> 예) `feature/notice/list-search` → 도메인 `notice`, 기능 `list-search`

| 브랜치명 | feature-path | 문서 경로 |
|---------|-------------|---------|
| `feature/tag` | `tag` | `/src/features/tag/docs/` |
| `feature/notice/list` | `notice/list` | `/src/features/notice/list/docs/` |
| `feature/notice/list-search` | `notice/list-search` | `/src/features/notice/list-search/docs/` |
| `feature/notice/tag-filter` | `notice/tag-filter` | `/src/features/notice/tag-filter/docs/` |

**변환 알고리즘:**
1. `feature/` prefix 제거
2. 나머지를 그대로 `feature-path`로 사용 (슬래시·하이픈 변환 없음)

**Step 4: 추론 결과 사용자에게 확인**
```
🌿 현재 브랜치: feature/notice/list
📁 추론된 경로: notice/list
   → 문서 경로: /src/features/notice/list/docs/

경로가 맞으면 계속 진행합니다. 다르면 말씀해주세요.
```

**Step 5: 컨텍스트 저장**

경로 확정 후 세션 컨텍스트에 기록한다.
이후 `/tdd-red`, `/tdd-green`, `/tdd-refactor` 스킬은 이 값을 자동으로 사용한다.

```
[CONTEXT] feature-path: notice/list
          docs-root: /src/features/notice/list/docs/
          branch: feature/notice/list
```

> **이 컨텍스트가 있으면** `/tdd-red 1` 처럼 번호만 입력해도 경로를 자동으로 채운다.
> 컨텍스트가 없으면 TDD 스킬은 `/tdd-red {feature-path} 1` 형식을 요구한다.

## 경로 네이밍 규칙 (직접 지정 시)

직접 경로를 입력할 때의 형식 (브랜치 추론이 실패하거나 override할 때 사용).

```
/feature-planner {feature-path}
/feature-planner {feature-path} {기능 설명}
```

`{feature-path}`는 **기능의 위치를 나타내는 경로 전체**다.

| 입력 예시 | 의미 | 문서 경로 |
|----------|------|----------|
| `/feature-planner tag` | tag 단독 기능 | `/src/features/tag/docs/` |
| `/feature-planner notice/list` | notice 도메인의 list 기능 | `/src/features/notice/list/docs/` |
| `/feature-planner notice/list 페이지네이션 포함` | 설명 추가 | `/src/features/notice/list/docs/` |
| `/feature-planner notice/category/detail` | 3단계 중첩 | `/src/features/notice/category/detail/docs/` |

**파싱 규칙 (직접 지정 vs 브랜치 추론 구분):**
- 첫 토큰에 슬래시(`/`)가 있거나, 영문+하이픈 조합 → `{feature-path}` 직접 지정으로 판단
- 첫 토큰이 한글이거나 자연어 문장 → `{기능 설명}`으로 판단 → 브랜치 추론 실행
- 입력이 없음 → 브랜치 추론 실행

### 산출물 경로

```
/src/features/{feature-path}/docs/
```

## 파이프라인 개요

```
/feature-planner [{feature-path}] [{기능 설명}]
    ↓
feature-path 결정
  ├─ 직접 지정 → 그대로 사용
  └─ 브랜치 추론 → git branch --show-current
       ├─ main / master / develop / dev 감지
       │    └─ ⚠️ 보호 브랜치 경고 → 기능명 수집 → 브랜치명 제안
       │         → 사용자가 직접 git checkout -b 실행 → "완료" 응답 → 재확인
       └─ feature/* 감지 → 변환 → 사용자 확인
    ↓
[CONTEXT 저장: feature-path, docs-root, branch]
    ↓
spec.md 존재 확인 (/src/features/{feature-path}/docs/spec.md)
    ├─ 있음
    │    └─ 내용을 읽어 사용자에게 보여주고 "이 내용으로 진행할까요?" 확인 [GATE]
    │         ├─ 수정 요청 → 반영 후 재확인
    │         └─ 확인 → 진입 조건 체크리스트로 진행
    └─ 없음
         ├─ 기능 설명 있음 → spec.md 자동 생성 후 사용자 확인 [GATE]
         └─ 기능 설명 없음 → 최소 요구사항 인터뷰 (3문항) → spec.md 생성 후 확인 [GATE]
    ↓ 단계 0: 진입 조건 검증
    ↓ 단계 1: 요구사항 인터뷰 → spec-fixed.md [GATE]
    ↓ 단계 2: PRD + ADR 작성 → prd.md [GATE × 2]
    ↓ 단계 3: 이슈 분해 → task.md [GATE]
    ↓ /test-scenarios 핸드오프 (feature-path 자동 전달)
```

**산출물 경로**: `/src/features/{feature-path}/docs/`

## 단계 0: 진입 조건 검증

### 경로 확정 안내

스킬 진입 직후 파싱/추론 결과를 아래 형식으로 사용자에게 먼저 보여준다.
사용자가 수정을 요청하면 반영한 뒤 진행한다.

```
🌿 브랜치: feature/notice/list        ← 브랜치 추론 시에만 표시
📁 문서 경로: /src/features/notice/list/docs/

변경이 필요하면 말씀해주세요. 없으면 계속 진행합니다.
```

### spec.md 처리 — 3가지 경우 분기

#### 경우 A: spec.md가 이미 있음

파일을 읽어 내용을 사용자에게 그대로 보여주고 아래 메시지를 출력한다.

```
📄 기존 spec.md를 찾았습니다.
─────────────────────────────
{spec.md 내용}
─────────────────────────────
이 내용으로 진행할까요? 수정이 필요하면 말씀해주세요.
```

수정 요청이 오면 반영 후 재확인한다.
긍정 응답이 오면 진입 조건 체크리스트로 진행한다.

#### 경우 B: spec.md 없음 + 기능 설명 있음

호출 시 함께 전달된 설명으로 아래 형식의 `spec.md`를 즉시 생성하고 보여준다.
수정 요청이 오면 반영 후 재확인한다.
긍정 응답이 오면 진입 조건 체크리스트로 진행한다.

```markdown
# {기능명} 정의서

## 기능 개요
{설명에서 추출한 기능 목적. 한두 문장.}

## 기능 요구사항
{설명에서 추출한 사용자 행동 목록}

## 제약사항
{설명에서 추출한 제한 조건. 없으면 "없음".}
```

#### 경우 C: spec.md 없음 + 기능 설명도 없음

**한 번에 하나씩** 아래 3문항을 질문해 최소 요구사항을 수집한다.
빠른 진행을 위해 각 질문에 추천 예시를 함께 제시한다.

```
[질문 1] 이 기능의 목적이 무엇인가요?
예) "게시글에 태그를 붙여서 카테고리처럼 필터링에 활용"
```

```
[질문 2] 누가 사용하나요?
예) 일반 사용자 / 관리자 / 둘 다
```

```
[질문 3] 핵심 동작을 한 줄로 표현하면?
예) "태그 추가·삭제·목록 조회"
```

3문항 답변이 완료되면 경우 B와 동일하게 `spec.md`를 생성하고 사용자에게 확인받는다.

### 진입 조건 체크리스트

아래 항목이 모두 충족되어야 단계 1을 시작한다.

- [ ] 기능 개요가 작성되어 있다
- [ ] 요구사항이 1개 이상 존재한다
- [ ] 제약사항이 작성되어 있다

## 단계 1: 요구사항 인터뷰

### 목적

spec.md의 모호성을 제거하고 엣지 케이스·결정 사항을 확정한다.
인터뷰 전에 코드베이스를 탐색해 아래 두 가지를 파악하고 결과를 기록한다.

**공통 컴포넌트 인벤토리**
공통 컴포넌트 경로는 코드베이스를 먼저 탐색해 확인한다 (예: `src/components/common/`).

| 컴포넌트 | 이 기능에서 재사용 가능성 |
|----------|--------------------------|
| {컴포넌트명} | 높음 / 낮음 / 해당없음 |

**디자인 토큰 현황**
디자인 토큰 경로는 코드베이스를 먼저 탐색해 확인한다 (예: `src/styles/`).
- 사용 가능한 색상 토큰 목록
- 사용 가능한 스페이싱/타이포그래피 토큰 목록

**Zustand Store 현황**
기존 Store 파일을 탐색해 현재 상태 구조를 파악한다.
- 기존 Store 목록과 관리 중인 상태
- 이 기능에서 추가·수정이 필요한 Store 영역

이 목록이 인터뷰 질문 7번과 PRD "디자인 시스템 준수 계획" 및 "기술 결정" 섹션의 입력이 된다.

### 인터뷰 방식

**한 번에 하나의 질문만 한다.** 사용자가 답하면 다음 질문으로 넘어간다.
추천 방식이 있으면 이유와 함께 제시한다.
spec.md에 이미 명확히 답변된 영역은 건너뛴다.
모든 영역이 다뤄지면 인터뷰를 종료한다.

**질문 영역 (맥락 → 동작 → 기술 → 미래 순):**

[맥락]
1. 이 기능의 primary user는 누구인가?
2. 참고할 화면이나 레퍼런스가 있는가? (Figma, 스크린샷, 유사 서비스)
3. 의존하는 기존 기능은 무엇인가? (데이터 저장 방식과의 관계)

[동작 정의]
4. 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)
5. 경계 조건 — 최대값, 빈 값, 중복, 동시성은?
6. 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?

[기술 제약]
7. 기존 UI 패턴과의 일관성 — 공통 컴포넌트를 그대로 재사용할 수 있는가,
   아니면 확장이 필요한가? 새로 만들어야 한다면 그 이유는?
8. 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?

[미래]
9. 향후 확장 가능성 — 이 결정이 다음 feature에 영향을 주는가?

### 산출물: spec-fixed.md

인터뷰 내용을 반영해 `spec-fixed.md`를 작성한다.
반드시 아래 "용어 정의" 섹션을 포함한다. 이 용어표가 이후 모든 코드·이슈·PR·커밋 메시지의 기준이 된다.

```markdown
## 용어 정의

| 용어 | 설명 | 코드 네이밍 |
|------|------|------------|
| {용어} | {설명} | `camelCase` / `PascalCase` |
```

**용어 정의 규칙:**
- 타입명: PascalCase (예: `Tag`, `TagList`)
- 함수명: camelCase 동사+명사 (예: `addTag`, `removeTag`)
- 컴포넌트명: PascalCase (예: `TagBadge`, `TagInputField`)
- Zustand Store 액션명: camelCase 동사+명사 (예: `addTag`, `removeTag`, `setTags`)
- 동의어 사용 금지 — tag/label/keyword 중 하나를 선택하면 나머지는 코드에서 쓰지 않는다
- 용어가 바뀌어야 한다면 spec-fixed.md를 먼저 수정하고 코드를 따라 바꾼다

### [GATE] 단계 1 → 단계 2

```
[GATE] spec-fixed.md를 사용자에게 보여주고
       긍정 응답(yes / ok / 확인 / 좋아)을 받을 때까지 대기.
       긍정 응답 전까지 단계 2를 시작하지 않는다.
```

---

## 단계 2: PRD 작성 + 기술 결정(ADR)

### 목적

요구사항·디자인·기술 결정·범위를 `prd.md` 하나로 통합한다.
"이 기능에 대해 궁금하면 여기만 보면 된다"는 단일 기준점을 만드는 것이 목표다.

### 2-1. PRD 뼈대 생성

기술 결정 섹션은 TODO로 비워두고 나머지를 먼저 작성한다.

```markdown
# {기능명} PRD

## 개요
{기능 목적과 범위 — 한 단락}

## 사용자 스토리
- As a {user}, I want to {action} so that {benefit}

## 디자인 시스템 준수 계획

### 공통 컴포넌트 사용 원칙
> **공통 컴포넌트는 절대 수정하지 않는다.**
> 그대로 사용하거나, 기능이 부족하면 피처 컴포넌트로 래핑·확장해서 사용한다.

| 공통 컴포넌트 | 사용 방식 | 사용 위치 |
|--------------|----------|----------|
| {컴포넌트명} | 그대로 사용 / 래핑 확장 | {어느 UI에서 사용} |

### 신규 생성이 필요한 컴포넌트
공통 컴포넌트를 래핑·확장해도 구현이 불가능한 경우에만 신규 생성한다.

| 컴포넌트 | 공통 컴포넌트로 해결 불가한 이유 |
|----------|--------------------------------|
| {컴포넌트명} | {이유} |

- 색상: CSS 변수만 사용. Tailwind 기본 팔레트 직접 사용 금지.
- 스페이싱/타이포그래피: 프로젝트 토큰만 사용. 하드코딩 px/hex/rgb 금지.
```

### 2-2. 아키텍처 3안 제안 & 비교

코드베이스의 기존 패턴을 먼저 탐색한 뒤, 최소 3가지 구현 안을 제안하고
아래 **8가지 고정 기준**으로 비교표를 작성한다.

| # | 기준 | 설명 |
|---|------|------|
| 1 | 데이터 구조 | 어떤 형태로 데이터를 저장/표현하는가 |
| 2 | API 레이어 변경지점 | 기존 API에서 무엇이 바뀌는가 |
| 3 | Zustand Store 변경지점 | 기존 Store에서 무엇이 추가·수정되는가 |
| 4 | 핵심 동작 | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5 | 컴포넌트 구조 | 재사용할 공통 컴포넌트와 신규 생성이 불가피한 컴포넌트를 구분한다 |
| 6 | 기존 패턴과의 일관성 | 코드베이스 관습과 얼마나 맞는가 |
| 7 | 테스트 용이성 | 단위/E2E 테스트 작성이 쉬운가 |
| 8 | 구현 복잡도 / 예상 공수 | 팀 역량과 일정을 고려했을 때 현실적인가 |

> **8번 기준을 포함하는 이유:** 가장 우아한 아키텍처가 항상 최선은 아니다.
> 팀 역량과 일정을 반영해야 실제로 완성된다.

```
[GATE] 사용자가 3안 중 하나를 선택할 때까지 대기.
       선택 전까지 2-3을 시작하지 않는다.
```

### 2-3. ADR 작성 (PRD "기술 결정" 섹션에 삽입)

선택한 안을 아래 4요소 형식으로 기록한다.
이유 없는 거부와 단점을 숨긴 결과는 미래의 의사결정자를 오도한다.

```markdown
### {결정 제목}

**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 각각의 거부 이유. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. 장점뿐 아니라 단점도 반드시 명시한다.
```

### 2-4. Out of Scope 작성

"이번에 하지 않을 것"을 구체적으로 나열한다.
Out of Scope가 없으면 AI가 맥락을 확장 해석해 범위 초과 구현이 발생한다.
예: "태그 색상 커스터마이징", "태그 자동 추천", "태그 기반 검색" 등

```
[GATE] Out of Scope를 사용자에게 보여주고
       긍정 응답(yes / ok / 확인 / 좋아)을 받을 때까지 대기.
       긍정 응답 전까지 단계 3을 시작하지 않는다.
```

### 산출물

- `/src/features/{feature-path}/docs/prd.md`

## 단계 3: 이슈 분해

### 목적

PRD를 실행 가능한 작업 단위로 변환한다.
각 이슈는 독립적으로 TDD 사이클(Red → Green → Refactor)을 돌릴 수 있어야 한다.

### 수직 슬라이싱 원칙

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- **Yes** → 올바른 수직 슬라이스
- **No** → 다른 이슈와 합치거나 다시 나눈다

**수평 슬라이싱 금지** (레이어별 분리: API만 → Store만 → UI만).
수평 슬라이싱은 마지막 이슈까지 완료해야 동작이 나오므로 이슈 단위 TDD를 돌릴 수 없다.

### 이슈 크기 기준

**완료 시간**: Red → Green → Refactor 사이클을 반나절~하루 안에 완료 가능해야 한다.

**테스트 가능성 — FIRST 원칙**

| 원칙 | 기준 |
|------|------|
| Fast | 테스트가 수 초 안에 실행 |
| Isolated | 다른 이슈 없이 단독 실행 가능 |
| Repeatable | 환경과 무관하게 결과 동일 |
| Self-validating | 통과/실패가 자동 판단 |
| Timely | 구현 직전에 테스트를 먼저 작성할 수 있는 단위 |

**코드 변경 범위 — 보조 지표** (초과 시 재분해)

| 항목 | 기준 |
|------|------|
| 변경 파일 수 | 5개 이하 |
| 신규 컴포넌트 수 | 2개 이하 |
| AC 항목 수 | 5개 이하 |

### 자가 검증 (이슈 작성 후 반드시 확인)

- "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?" → **Yes**여야 진행
- "이 이슈의 테스트를 앞 이슈 없이 단독으로 작성할 수 있는가?" → **Yes**여야 진행
- "이 이슈에서 신규 생성하는 컴포넌트가 공통 컴포넌트를 그대로 쓰거나 래핑·확장해서 해결 가능한가?" → **Yes**라면 공통 컴포넌트 활용으로 이슈 수정. **No**(공통으로 해결 불가)일 때만 신규 생성을 허용한다. 공통 컴포넌트 자체는 수정 금지.
- "이 이슈의 UI 구현에 하드코딩 색상/스페이싱이 포함되어 있는가?" → **No**여야 진행

### Acceptance Criteria 형식

각 이슈에 반드시 Given-When-Then 형식의 AC를 포함한다.
이 AC가 `/test-scenarios` 스킬의 테스트 시나리오 입력이 된다.

```markdown
## Acceptance Criteria
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
```

### 의존성 순서

앞 이슈의 결과가 다음 이슈의 입력이 되도록 배치한다. 역방향 개발 금지.

### 네이밍 규칙

이슈 제목·AC·구현 코드 모두 `spec-fixed.md`의 용어 정의를 따른다.
동의어 사용 금지 — 용어가 바뀌어야 한다면 spec-fixed.md를 먼저 수정하고 코드를 바꾼다.

### [GATE] 최종 승인

```
[GATE] task.md를 사용자에게 보여주고 긍정 응답을 받을 때까지 대기.
       확인 항목:
         - 수직 슬라이스 기준을 충족하는가
         - AC가 Given-When-Then 형식으로 구체적으로 작성되었는가
         - 의존성 순서가 올바른가
       긍정 응답 전까지 GitHub Issues 등록 등 외부 작업을 진행하지 않는다.
```

### 산출물

- `/src/features/{feature-path}/docs/task.md`
- 프로젝트 칸반 보드 Todo 컬럼에 배치

### 핸드오프

task.md 확정 후 `/test-scenarios` 스킬로 이관한다.
각 이슈의 AC(Given-When-Then)가 `/test-scenarios`에서 `issue-{N}.md` 생성의 입력이 된다.

**TDD 스킬로의 컨텍스트 전달:**
이 스킬에서 확정된 `[CONTEXT]`가 살아있는 동안 TDD·test-scenarios 스킬은 경로 입력을 요구하지 않는다.
```
다음 단계: /test-scenarios  (feature-path 생략 가능 — 컨텍스트에서 자동 로드)
           /tdd-red 1       (feature-path 생략 가능 — 컨텍스트에서 자동 로드)
```

## TDD · test-scenarios 스킬 연동 규칙

### 컨텍스트 유효 범위

`/feature-planner`가 세션에서 실행되어 `[CONTEXT]`가 기록된 경우:

| 스킬 호출 | 동작 |
|----------|------|
| `/test-scenarios` | 컨텍스트의 feature-path + task.md 자동 로드 |
| `/tdd-red 1` | 컨텍스트의 feature-path + 이슈 번호 1 사용 |
| `/tdd-green 2` | 컨텍스트의 feature-path + 이슈 번호 2 사용 |
| `/tdd-refactor 1` | 컨텍스트의 feature-path + 이슈 번호 1 사용 |
| `/test-scenarios notice/list` | 명시적 경로 우선 (컨텍스트 무시) |
| `/tdd-red notice/list 1` | 명시적 경로 우선 (컨텍스트 무시) |

### 컨텍스트가 없을 때

TDD 스킬을 단독으로 실행하는 경우:
```
/tdd-red {feature-path} {이슈 번호}
/tdd-red notice/list 1
```

### 브랜치가 이미 `feature/xxx` 형태라면

TDD 스킬도 `/tdd-red`만 입력 시 현재 브랜치에서 자동 추론을 시도한다.
단, feature-planning 컨텍스트가 있으면 git 추론보다 컨텍스트를 우선한다.

## 승인 게이트 요약

| 지점 | 확인 내용 | 긍정 응답 예시 |
|------|----------|--------------|
| 단계 0 진입 | 브랜치 추론 경로 확정 | yes / ok / 확인 / 좋아 또는 수정 요청 |
| 단계 0 — spec.md 확인 | 기존 spec 검토 또는 신규 생성 확인 | yes / ok / 확인 / 좋아 또는 수정 요청 |
| 단계 0 후 | 진입 조건 체크리스트 자동 확인 | — (자동 진행) |
| 단계 1 후 | spec-fixed.md — 모호성 제거, 용어 확정 | yes / ok / 확인 / 좋아 |
| 단계 2-2 후 | 아키텍처 3안 중 하나 선택 | "2안으로 진행" 등 |
| 단계 2-4 후 | Out of Scope — 범위 확정 | yes / ok / 확인 / 좋아 |
| 단계 3 후 | 이슈 목록 — 수직슬라이스·AC·의존성 확인 | yes / ok / 확인 / 좋아 |

GATE가 없으면 AI가 판단 없이 끝까지 달려버린다.
각 GATE에서 반드시 멈추고 사용자의 응답을 기다릴 것.
