# 기획 워크플로우

하나의 "아이디어"에서 "개발 가능한 이슈 단위"로 변환하는 파이프라인.

## 진입점

```
/feature-planner 
    ↓
spec.md 존재 여부 확인
    ├─ 없음 → spec.md 자동 생성 → 단계 0
    └─ 있음 → 단계 0
```

## 전체 흐름

```
spec.md (초기 아이디어)
    ↓ 단계 0 — 진입 조건 검증
    ↓ 단계 1 — 요구사항 인터뷰
spec-fixed.md (확정된 요구사항 + 용어 정의)
    ↓ 단계 2 — PRD 작성 + 기술 결정(ADR)
prd.md (단일 기준 문서)
    ↓ 단계 3 — 이슈 분해
issues.md (실행 가능한 작업 단위)
    ↓ 핸드오프
/test-scenarios (별도 스킬에서 작성)
```

---

## 단계 0: 진입 조건 검증

### 목적
인터뷰를 시작하기에 최소한의 내용이 갖춰졌는지 확인한다.
아래 항목이 모두 존재해야 단계 1을 시작한다. 모호한 내용은 단계 1에서 채운다.

### spec.md가 없는 경우 — 자동 생성

`/feature-planner {요구사항}` 실행 시 `spec.md`가 없으면,
전달받은 요구사항을 바탕으로 아래 형식으로 spec.md를 생성한 뒤 진입 조건 체크리스트를 확인한다.

```markdown
# {기능명} 정의서

## 기능 개요
{요구사항에서 추출한 기능 목적. 한두 문장.}

## 기능 요구사항
{요구사항에서 추출한 사용자 행동 목록}

## 제약사항
{요구사항에서 추출한 제한 조건. 없으면 "없음".}
```

> spec.md 생성 후 사용자에게 내용을 보여주고 수정이 필요한지 확인한다.
> 수정이 없으면 진입 조건 체크리스트로 자동 진행한다.

### 진입 조건 체크리스트

- [ ] 기능 개요가 작성되어 있다
- [ ] 요구사항이 1개 이상 존재한다
- [ ] 제약사항이 작성되어 있다

---

## 단계 1: 요구사항 인터뷰

### 목적
초기 기능 정의서의 모호성을 제거하고, 엣지 케이스와 결정 사항을 확정한다.

### 입력
- `/src/features/{name}/docs/spec.md`
- 코드베이스 (기존 구조·패턴 참조)

### 핵심 활동

**AI를 인터뷰어로 활용한다.** 프롬프트 패턴:

```
이 정의서를 바탕으로, 코드베이스를 탐색하고 구현 전에 확인이 필요한 것들을
나에게 질문해라. 한 번에 하나씩 질문하고 내가 답을 하면 다음 질문으로 넘어간다.
추천 방식도 이유와 함께 포함하라.
미결 사항에 적힌 항목은 중복으로 질문하지 않고 답변을 참고해 다음 질문을 이어가라.
```
**인터뷰 전 준비:**
spec.md를 분석해 미결 사항을 파악하고, 해당 항목을 아래 영역 중 적절한 곳에 끼워 넣어 질문한다.
spec.md에 이미 명확히 답변된 영역은 건너뛴다.
모든 영역이 다뤄졌을 때 인터뷰를 종료한다.

**질문 영역 — 맥락 → 동작 → 기술 제약 → 미래 순으로 진행:**

[맥락]
1. 이 기능의 primary user는 누구인가?
2. 참고할 화면이나 레퍼런스가 있는가? (Figma, 스크린샷, 유사 서비스)
3. 이 기능이 의존하는 기존 기능은 무엇인가? (데이터 저장 방식과의 관계)

[동작 정의]
4. 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)
5. 경계 조건 — 최대값, 빈 값, 중복, 동시성은?
6. 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?

[기술 제약]
7. 기존 UI 패턴과의 일관성 — 공통 컴포넌트를 재사용할 수 있는가?
8. 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?

[미래]
9. 향후 확장 가능성 — 이 결정이 다음 feature에 영향을 주는가?

### Ubiquitous Language 정의

인터뷰 과정에서 사용되는 용어를 고정한다.
`spec-fixed.md`에 "용어 정의" 섹션으로 명시하며, 아래 형식을 따른다.

```markdown
## 용어 정의

| 용어 | 설명 | 코드 네이밍 |
|------|------|------------|
| 태그 | 노트에 붙이는 문자열 식별자 | `tag` / `Tag` |
| 태그 목록 | 노트에 연결된 태그의 배열 | `tags` / `TagList` |
| 태그 추가 | 노트에 태그를 연결하는 행위 | `addTag` |
| 태그 삭제 | 노트에서 태그를 제거하는 행위 | `removeTag` |
```

**코드 네이밍 규칙:**
- 타입명: PascalCase로 용어 그대로 사용 (예: `Tag`, `TagList`)
- 함수명: camelCase 동사+명사 (예: `addTag`, `removeTag`, `fetchTags`)
- 컴포넌트명: PascalCase (예: `TagBadge`, `TagInputField`)
- 동의어 사용 금지 — `label`, `keyword`, `category` 등 유사 단어를 코드에서 쓰지 않는다
- 용어가 바뀌어야 한다면 spec-fixed.md를 먼저 수정하고 코드를 따라 바꾼다
- 변수명·함수명·컴포넌트명·타입명·PR 제목·커밋 메시지 모두 이 용어를 그대로 사용한다

### 산출물

- `/src/features/{name}/docs/spec-fixed.md`

### 승인 게이트

```
[GATE] 사용자가 spec-fixed.md를 읽고 긍정 응답(yes / ok / 확인 / 좋아 등)을 할 때까지 대기.
긍정 응답 전까지 단계 2를 시작하지 않는다.
```

## 단계 2: PRD 작성 + 기술 결정(ADR)

### 목적
요구사항·디자인·기술 결정·범위를 하나의 문서로 통합해
"이 기능에 대해 궁금하면 여기만 보면 된다"는 단일 기준점을 만든다.

### 입력
- `/src/features/{name}/docs/spec-fixed.md`
- 코드베이스 (기존 패턴 참조)

### 핵심 활동

**2-1. PRD 뼈대 생성** (기술 결정은 TODO로 비워둠)

PRD 구조:
- **개요** — 기능의 목적과 범위
- **사용자 스토리** — 사용자 관점의 흐름
- **기술 결정** — 어떤 구조 위에 올릴 것인가 (2-3에서 ADR 작성)
- **Out of Scope** — 이번에 하지 않을 것
- **용어 정의** — spec-fixed.md의 Ubiquitous Language 동기화

---

**2-2. 아키텍처 3안 제안 & 비교**

AI에게 최소 3가지 안을 제안받고, **아래 8가지 고정 기준**으로 비교한다.

| # | 기준 | 설명 |
|---|------|------|
| 1 | 데이터 구조 | 어떤 형태로 데이터를 저장/표현하는가 |
| 2 | API 레이어 변경지점 | 기존 API에서 무엇이 바뀌는가 |
| 3 | 상태관리 변경지점 | Context/Store에서 무엇이 추가되는가 |
| 4 | 핵심 동작 | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5 | 컴포넌트 구조 | 어떤 컴포넌트가 추가/수정되는가 |
| 6 | 기존 패턴과의 일관성 | 코드베이스의 기존 관습과 얼마나 맞는가 |
| 7 | 테스트 용이성 | 단위 테스트, E2E 테스트 작성이 쉬운가 |
| 8 | 구현 복잡도 / 예상 공수 | 팀 역량과 일정을 고려했을 때 현실적으로 가능한가 |

> **8번 기준을 추가한 이유:** 아키텍처 선택은 기술적 우열만이 아니라 팀 역량과 일정 현실도 반영해야 한다. 가장 우아한 안이 항상 최선은 아니다.

비교 결과를 표로 정리한 뒤 **사용자가 선택**한다.

```
[GATE] 사용자가 3안 중 하나를 선택하는 긍정 응답을 할 때까지 대기.
긍정 응답 전까지 2-3을 시작하지 않는다.
```

---

**2-3. 선택한 안을 ADR 4요소로 PRD "기술 결정" 섹션에 작성**

```markdown
### 기술 결정 제목

**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 각각의 거부 이유. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. 장점뿐 아니라 단점도 반드시 명시한다.
```

---

**2-4. Out of Scope 작성**

"하지 않을 것"을 구체적으로 나열한다.

> **Out of Scope를 명시해야 하는 이유:** 명시하지 않으면 AI가 맥락을 확장 해석해
> 범위 초과 구현을 한다. 예: "태그 색상 커스터마이징", "태그 자동 추천", "태그 기반 검색" 등

```
[GATE] 사용자가 Out of Scope를 읽고 긍정 응답을 할 때까지 대기.
긍정 응답 전까지 단계 3을 시작하지 않는다.
```

### 산출물

- `/src/features/{name}/docs/prd.md`

---

## 단계 3: 이슈 분해

### 목적
PRD를 실행 가능한 작업 단위로 변환한다.
각 이슈는 독립적으로 TDD 사이클을 돌릴 수 있어야 한다.

### 입력
- `/src/features/{name}/docs/prd.md`

### 핵심 활동

**수직 슬라이싱 원칙:**

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- **Yes** → 수직 슬라이스. TDD 사이클을 돌릴 수 있다.
- **No** → 다른 이슈와 합치거나 다시 나눈다.

수평 슬라이싱(레이어별: API → Context → UI)은 **금지**.
마지막 이슈까지 완료해야 동작이 나오므로 이슈 단위 TDD 사이클을 돌릴 수 없다.

---

**이슈 크기 기준 (TDD 모범 사례)**

하나의 이슈는 아래 조건을 모두 충족해야 한다.

**완료 시간**
- Red → Green → Refactor 사이클을 반나절~하루 안에 완료 가능할 것

**테스트 가능성 — FIRST 원칙**

| 원칙 | 기준 |
|------|------|
| Fast | 테스트가 수 초 안에 실행된다 |
| Isolated | 다른 이슈·모듈에 의존하지 않고 단독 실행된다 |
| Repeatable | 환경에 상관없이 결과가 동일하다 |
| Self-validating | 통과/실패가 자동으로 판단된다 |
| Timely | 구현 직전에 테스트를 먼저 작성할 수 있는 단위다 |

**코드 변경 범위 — 보조 지표**

| 항목 | 기준 |
|------|------|
| 변경 파일 수 | 5개 이하 |
| 신규 컴포넌트 수 | 2개 이하 |
| AC 항목 수 | 5개 이하 |

보조 지표를 초과하면 이슈를 다시 분해한다.

**분해 판단 질문 (이슈 작성 후 자가 검증)**
- "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?" → **Yes**여야 진행
- "이 이슈의 테스트를 앞 이슈 없이 단독으로 작성할 수 있는가?" → **Yes**여야 진행

---

**의존성 순서:**
- 앞 이슈의 결과가 다음 이슈의 입력이 되도록 배치
- 역방향 개발 금지

---

**코드 네이밍 규칙 (이슈 작성 시 적용)**
- 이슈 제목, AC, 구현 코드 모두 `spec-fixed.md`의 용어 정의를 따른다
- 동의어 사용 금지 — 용어가 바뀌어야 한다면 `spec-fixed.md`를 먼저 수정한다

---

**Acceptance Criteria — Given-When-Then 형식:**

```markdown
## Acceptance Criteria
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
```

각 이슈에 반드시 AC를 포함한다.
AC는 다음 단계(`/test-scenarios`)에서 테스트 시나리오의 씨앗이 된다.

---

```
[GATE] 사용자가 이슈 목록을 읽고 긍정 응답을 할 때까지 대기.
긍정 응답 전까지 GitHub Issues 등록을 진행하지 않는다.
확인 항목:
  - 수직 슬라이스 기준을 충족하는가
  - AC가 Given-When-Then 형식으로 구체적으로 작성되었는가
  - 의존성 순서가 올바른가
```

### 산출물
- `/src/features/{name}/docs/issues.md`
- 프로젝트 칸반 보드 Todo 컬럼에 배치

### 다음 단계
issues.md 확정 후 → `/test-scenarios` 스킬로 이관.
각 이슈의 AC(Given-When-Then)가 테스트 시나리오의 입력이 된다.

---

## 승인 게이트 요약

| 지점 | 확인 내용 | 긍정 응답 예시 |
|------|----------|--------------|
| 단계 0 후 | 기능 개요·요구사항·제약사항 존재 확인 | — (자동 진행) |
| 단계 1 후 | spec-fixed.md — 모호성 제거, 용어 확정 | yes / ok / 확인 / 좋아 |
| 단계 2-2 후 | 아키텍처 3안 중 하나 선택 | "2안으로 진행" 등 |
| 단계 2-4 후 | Out of Scope — 범위 확정 | yes / ok / 확인 / 좋아 |
| 단계 3 후 | 이슈 목록 — 수직 슬라이스, AC 구체성, 의존성 순서 | yes / ok / 확인 / 좋아 |

> 이 게이트들이 없으면 AI가 판단 없이 끝까지 달려버린다.
> 스킬에 반드시 `[GATE]`로 명시해야 한다.
