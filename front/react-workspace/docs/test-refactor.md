skill-creator를 사용해서 아래 조건으로 .claude/skills/tdd-refactor/SKILL.md를 만들어줘:

## 목적
테스트 통과 상태를 유지하면서 코드 구조를 개선 (TDD Refactor 단계)


## feature-path 결정
1. feature-planner 세션 컨텍스트가 있으면 feature-path를 자동 로드해서 사용 하고 없으면
2. Git 브랜치명(feature/xxx)을 자동 파싱해 feature-path를 설정한다.
3. 그것도 없으면 직접 입력 받는다. (/test-refactor {feature-path} {N})

---

## 입력 스펙
- $ARGUMENTS = {이슈 번호} (필수)
- 예) /tdd-green 42  →  src/features/{feature-path}/issue-42.md 를 읽음

---

## 실행 순서
1. CLAUDE.md 읽기 → 프로젝트별 컨벤션(네이밍, 패턴, 스타일링) 파악
2. pnpm test:run 실행 → 전체 통과 확인부터 시작
   - 실패 테스트가 있으면: 즉시 중단하고 tdd-green 먼저 실행을 사용자에게 요청
   - 전체 통과 확인 후 다음 단계로 이동
3. 리팩토링 대상 파일 식별
    - `git diff main...HEAD --name-only -- src/` 로
      현재 브랜치에서 변경된 src/ 내 파일 목록 추출
      (issue-{번호}.md 의 시그니처에 명시된 파일과 교차 검증하여 범위를 좁힘)
    - 테스트 파일(*.test.ts / *.test.tsx) 제외
    - 남은 파일을 아래 범용 점검 기준으로 검토
    - 범용 점검 기준 (스킬 내장):
       - 중복 제거: 같은 로직이 반복되는가
       - 네이밍 명확성: 의도를 드러내는 이름인가
       - 단일 책임: 하나의 함수/컴포넌트가 너무 많은 일을 하는가
       - 불필요한 복잡도: 매직 넘버, 과도한 중첩, 불필요한 추상화
       - CLAUDE.md 컨벤션 불일치: 프로젝트 패턴과 어긋나는 코드
4. 리팩토링 대상 목록을 개발자에게 보고 → 승인 대기
5. 승인된 항목에 대해 하나씩 리팩토링 수행 〈피드백 루프〉
    - 하나의 변경 → pnpm test:run 실행 → 전체 통과 확인 → 다음 항목으로 이동
    - 테스트가 깨지면(빨간): 즉시 롤백 후 다른 접근 방식으로 재시도
    - 재시도 후에도 깨지면: 해당 항목을 건너뛰고 사용자에게 보고
7. 전체 완료 후 결과 요약 출력
    - 리팩토링 전후 변경 항목 목록
    - 각 변경의 개선 이유 한 줄 요약
    - 건너뛴 항목이 있으면 별도 표시

## 제약
- 새 기능 추가 금지
- 테스트 파일 수정 금지
- src/features/{feature-path}/ 디렉토리 밖 수정 금지
- git diff 결과 대상 파일 범위 밖 수정 금지
