# ADR-0002: PR 머지 전략 및 blocked-by 이슈의 브랜치 전략

- **상태(Status)**: 수락됨(Accepted) — 2026-07-16
- **날짜**: 2026-07-16 (merge commit → rebase merge로 재전환, 실무 관행 반영)
- **컨텍스트**: Git 워크플로우 / 멀티 이슈 구현

## 배경(Context)

`/to-issues`로 분해된 이슈들은 서로 `blocked by` 관계를 가질 수 있다(예: 이슈 B가 이슈 A의 컴포넌트를 재사용). 이슈 #36(로그인 화면) → #37(401 처리), #38(스켈레톤 페이지) 구현 중, #37/#38을 #36의 PR이 머지되기 **전에** #36의 기능 브랜치(`feat/36-login-page`) 위에 스택으로 얹어 작업한다.

2026-07 리팩토링 시리즈(#67, #69)에서 실제 머지가 일관되게 **rebase merge**로 수행되었다 — 문서(merge commit)와 실무가 어긋나 있었으므로, 실무를 기준으로 문서를 개정한다.

## 결정(Decision)

1. **머지 전략은 rebase merge**. PR 병합 시 "Rebase and merge" 버튼을 사용한다. 개별 커밋이 `main`에 반영되되 SHA는 재작성되며, `main` 히스토리는 선형으로 유지된다.
2. **blocked-by 이슈도 선행 PR 머지 대기 없이 선행 브랜치 위에 스택해서 진행할 수 있다.**
3. **선행 PR 머지 후 스택 브랜치는 `git rebase main`으로 갱신한다.** rebase merge로 SHA가 재작성되었어도 git이 patch-id로 중복 커밋을 감지해 대부분 자동으로 정리된다. 충돌 시 수동 해결한다.
4. **PR 머지 확인 후 브랜치를 정리한다.** SHA 재작성으로 조상 관계가 보존되지 않으므로 `git branch -d`가 경고할 수 있다 — **원격 머지를 확인한 뒤** `git branch -D <브랜치>`를 사용한다. 원격 브랜치도 함께 정리한다.

## 결과(Consequences)

- **긍정**: `main` 히스토리가 선형으로 유지되어 로그 가독성이 좋다. blocked-by 체인의 병렬화는 계속 가능하다.
- **부정/비용**: 원본 SHA가 재작성되어 로컬 브랜치와 `main`의 조상 관계가 끊긴다 — 브랜치 삭제에 `-D`가 필요할 수 있고, 스택 브랜치 rebase 시 patch-id 감지에 실패하면(머지 과정에서 내용이 바뀐 경우) 수동 정리가 필요하다.

---

## 변경 이력

1. **최초(2026-07-07)**: Squash merge — `main` 히스토리를 PR당 커밋 하나로 깔끔하게 유지하지만, blocked-by 이슈 스택 시 전체 파일 충돌 문제가 발생.
2. **개정(2026-07-08)**: Rebase merge — 원본 커밋이 `main`에 반영되는 것처럼 동작하지만, 실제로는 SHA가 재작성되어 조상 관계가 보존되지 않음. 복잡한 규칙 필요.
3. **개정(2026-07-10)**: Merge commit — 원본 커밋이 그대로 보존되어 조상 관계가 완전히 유지됨. 단순하고 명확한 규칙.
4. **최종(2026-07-16)**: Rebase merge로 재전환 — #67·#69가 실제로 rebase merge로 병합되어 문서와 실무가 불일치. 선형 히스토리 선호가 확인되어 실무를 기준으로 개정. 스택 브랜치·브랜치 삭제의 비용은 3·4항의 규칙으로 흡수.
