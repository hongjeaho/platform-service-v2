# ADR-0002: PR 머지 전략 및 blocked-by 이슈의 브랜치 전략

- **상태(Status)**: 수락됨(Accepted) — 2026-07-10
- **날짜**: 2026-07-10 (rebase merge → merge commit로 전환)
- **컨텍스트**: Git 워크플로우 / 멀티 이슈 구현

## 배경(Context)

`/to-issues`로 분해된 이슈들은 서로 `blocked by` 관계를 가질 수 있다(예: 이슈 B가 이슈 A의 컴포넌트를 재사용). 이슈 #36(로그인 화면) → #37(401 처리), #38(스켈레톤 페이지) 구현 중, #37/#38을 #36의 PR이 머지되기 **전에** #36의 기능 브랜치(`feat/36-login-page`) 위에 스택으로 얹어 작업한다.

## 결정(Decision)

1. **머지 전략은 merge commit**. PR을 병합할 때 "Merge pull request" 버튼 또는 `git merge --no-ff`를 사용하여 머지 커밋을 생성한다. 원본 커밋이 `main`에 그대로 반영된다.
2. **blocked-by 이슈도 선행 PR 머지 대기 없이 선행 브랜치 위에 스택해서 진행할 수 있다.** 선행 브랜치가 `main`에 병합된 후에도 스택 브랜치는 정상적으로 병합할 수 있다.
3. **선행 PR 머지 후 스택 브랜치는 `git rebase main`으로 갱신한다.** 최신 `main` 위에서 재적용하여 충돌을 조기에 발견하고 해결한다.
4. **PR 머지 확인 후 `git branch -d <브랜치>`로 브랜치를 정리한다.** Merge commit 후 원본 커밋이 `main`의 실제 조상이 되므로 일반적인 삭제 명령이 동작한다. 원격 브랜치도 함께 정리한다.

## 결과(Consequences)

- **긍정**: blocked-by 체인을 선행 PR 머지 대기 없이 이어서 진행할 수 있어 병렬화 속도가 개선된다. 히스토리가 완전히 보존되어 재작성(rebase)으로 인한 문제가 없다. 브랜치 삭제가 정상적으로 동작한다.
- **부정/비용**: `main` 로그에 머지 커밋이 생성되어 히스토리가 선형이 아니다(단, 이는 Git의 기본 워크플로우이자 장점이기도 함).

---

## 변경 이력

1. **최초(2026-07-07)**: Squash merge — `main` 히스토리를 PR당 커밋 하나로 깔끔하게 유지하지만, blocked-by 이슈 스택 시 전체 파일 충돌 문제가 발생.
2. **개정(2026-07-08)**: Rebase merge — 원본 커밋이 `main`에 반영되는 것처럼 동작하지만, 실제로는 SHA가 재작성되어 조상 관계가 보존되지 않음. 복잡한 규칙 필요.
3. **최종(2026-07-10)**: Merge commit — 원본 커밋이 그대로 보존되어 조상 관계가 완전히 유지됨. 단순하고 명확한 규칙.
