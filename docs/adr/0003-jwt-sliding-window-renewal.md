# ADR-0003: JWT 세션 갱신 — 리프레시 토큰이 아닌 슬라이딩 윈도우 헤더 갱신

- **상태(Status)**: 수락됨(Accepted)
- **날짜**: 2026-07-09
- **컨텍스트**: 인증 / 세션 관리

## 배경(Context)

이 시스템은 별도의 리프레시 토큰이 없다. 로그인 시 JWT 하나(기본 24시간 만료, `jwt.expiration.period`)만 발급되고, `JWTCheckFilter`(`common/web/.../filter/JWTCheckFilter.java:75-93`)가 인증된 요청마다 남은 유효시간이 `renew-before`(기본 1시간, `jwt.expiration.renew-before`) 미만이면 새 토큰을 응답 `Authorization` 헤더에 실어 보내는 **슬라이딩 윈도우 방식**으로 세션을 연장한다. 토큰 회전(rotation)이나 서버 측 토큰 저장소(Redis/DB)는 없다 — 완전히 무상태(stateless)다.

이 설계는 어떤 ADR에도 기록된 적 없이 암묵적으로 존재해왔다. `/improve-codebase-architecture` 검토 중 다음 문제가 발견되었다: 프론트엔드(`front/react-workspace/src/api/client.ts`)는 로그인 응답의 `Authorization` 헤더만 명시적으로 읽고(`authService.ts:59`), 이후 모든 인증된 응답에서 서버가 실어 보내는 갱신 헤더는 전혀 읽지 않았다. 결과적으로 세션이 실질적으로 갱신되지 않고, 활동 중인 사용자도 24시간 뒤 강제 로그아웃되었다 — 서버가 준비한 갱신 신호가 클라이언트에서 조용히 버려지는 상태였다.

## 결정(Decision)

1. **리프레시 토큰/토큰 회전 방식으로 전환하지 않는다.** 리프레시 토큰 도입은 별도 서버 측 저장소와 회전·무효화 로직이 필요한 큰 변경이다. 이번엔 기존에 의도된 슬라이딩 윈도우 설계를 고치는 데 집중하고, 리프레시 토큰 전환 여부는 별도 논의로 미룬다.
2. **프론트엔드 응답 인터셉터가 갱신 헤더를 소비하도록 재연결한다.** `client.ts`의 응답 인터셉터가 인증된 응답의 `Authorization` 헤더에서 갱신된 토큰을 읽어 `authStore.renewToken(token)`으로 반영한다. `/api/public/**` 응답은 로그인 흐름과 개념이 겹치므로(로그인 응답의 헤더는 "갱신"이 아니라 최초 발급이며 이미 `authService.login()`이 처리) 이 갱신 소비 로직에서 제외한다.
3. **추출 로직은 순수 함수로 분리한다.** 헤더에서 갱신 토큰을 뽑아내는 판단(`extractRenewedToken`)은 axios 콜백과 분리된 순수 함수로 `client.ts`에 둔다 — 인터셉터는 이 함수를 호출하는 얇은 어댑터로 남는다.

이 ADR은 review candidate 1(갱신 seam 재연결)에 한정한다. 같은 검토에서 나온 candidate 2(`JWTCheckFilter`와 `SecurityConfig`의 "공개 경로" 판단 중복)와 candidate 3(`JWTCheckFilter`의 갱신 판단 로직을 서블릿 I/O에서 분리)은 이 ADR의 범위 밖이며 별도로 다룬다.

## 결과(Consequences)

- **긍정**: 활동 중인 사용자는 세션이 자연스럽게 연장되어 24시간마다 강제 로그아웃되지 않는다. 서버·클라이언트 양쪽에 흩어져 암묵적으로만 존재하던 설계가 한 문서에 기록되어, 향후 이 흐름을 다시 검토할 때 "왜 리프레시 토큰이 아닌가"를 처음부터 재조사할 필요가 없다.
- **부정/비용**: 여전히 리프레시 토큰이 없으므로 서버 측 토큰 강제 폐기가 불가능하다 — 로그아웃 시 클라이언트가 토큰을 지워도 서버는 그 토큰을 즉시 무효화할 방법이 없고, 탈취된 토큰은 자체 만료 시점까지(최대 24시간) 유효하다. 이 트레이드오프는 이번 결정의 범위 밖이며, 토큰 조기 무효화나 로그아웃 전파가 요구사항이 되면 별도 ADR로 리프레시 토큰 도입을 재검토한다.
